# Tape Sort - External Merge Sort на эмулированных ленточных накопителях

Консольное приложение на C++17, реализующее алгоритм внешней сортировки данных, хранящихся на эмулируемых ленточных накопителях, с жёстким ограничением доступной оперативной памяти.

## О проекте

Проект решает задачу сортировки произвольно большого объёма данных (N элементов типа `int32_t`), которые не помещаются в оперативную память, при ограничении памяти M байт. Алгоритм использует промежуточные ленточные файлы в директории `tmp/` и гарантирует, что объём используемой RAM никогда не превышает заданный лимит.

## Архитектура

Проект построен с соблюдением принципов чистой архитектуры и разделения ответственности:

| Компонент | Файл | Назначение |
|-----------|------|------------|
| ITape | `include/ITape.h` | Интерфейс ленточного накопителя |
| FileTape | `include/FileTape.h`, `src/FileTape.cpp` | Реализация интерфейса на основе бинарного файла |
| Config | `include/Config.h`, `src/Config.cpp` | Загрузка и хранение конфигурации |
| TapeSorter | `include/TapeSorter.h`, `src/TapeSorter.cpp` | Алгоритм внешней сортировки |
| main | `src/main.cpp` | Точка входа консольного приложения |

### Ключевые архитектурные решения

1. Абстракция через интерфейс `ITape`

Весь алгоритм сортировки написан исключительно в терминах абстрактного интерфейса `ITape`. Это позволяет:
- Легко подменять реализацию (файл, память, сетевой ресурс) без изменения сортировщика
- Тестировать алгоритм на моках и стабах
- Изолировать бизнес-логику от деталей хранения

2. Инкапсуляция конфигурации и временных задержек

Задержка каждого типа операции (чтение, запись, перемотка, сдвиг) настраивается независимо через внешний файл `config.txt` без перекомпиляции.
Для работы с временными интервалами используется value-object `Duration`, который представляет обёртку над различными представлениями времени. 

3. RAII

Деструктор `TapeSorter` автоматически удаляет все временные файлы, созданные в процессе сортировки. Даже при исключении все ресурсы корректно освобождаются.

## Алгоритм сортировки

Реализован классический сбалансированный 2-путевой внешний merge sort в две фазы:

### Фаза 1: Создание отсортированных прогонов

1. Из входной ленты считывается блок данных размером `M / sizeof(int32_t)` элементов - максимально возможный объём в рамках лимита памяти.
2. Блок сортируется в памяти с помощью `std::sort` (introsort, O(n log n)).
3. Отсортированный блок записывается во временный файл.
4. Процесс повторяется до исчерпания входных данных.

Сложность: O(N log(M/sizeof(int32_t))) — каждый элемент участвует в одной сортировке блока.

### Фаза 2: Итеративное слияние прогонов

1. Прогоны объединяются попарно: два отсортированных файла сливаются в один, при этом результат записывается в новый временный файл.
2. Слияние выполняется за один проход по обоим файлам (O(n + m)), без загрузки полных данных в память - элементы читаются по одному и сразу записываются.
3. После каждого раунда количество файлов уменьшается вдвое.
4. Процесс повторяется, пока не останется один файл - итоговый отсортированный результат.
5. Результат копируется на выходную ленту, временные файлы удаляются.

Сложность: O(N log R), где R — количество прогонов. Каждый элемент проходит log₂(R) раундов слияния.

### Итоговая сложность

- Время: O(N log N) - оптимально для сравнительной сортировки
- Память: O(M) - строго ограничено конфигурацией
- Временные файлы: O(log N) файлов в директории `tmp/`

## Оптимизации

1. Предварительное выделение файлов. Перед записью отсортированных данных временные файлы предварительно заполняются нулями до нужного размера. Это исключает системные вызовы расширения файла в процессе слияния и снижает фрагментацию на диске.

2. Кэширование значений при слиянии. Алгоритм merge читает текущий элемент из каждой ленты в локальную переменную и сравнивает кэшированные значения, а не обращается к ленте при каждом сравнении. Это сокращает количество операций seek/read на 50%.

3. Утилизация временных файлов. После каждого раунда слияния файлы, которые больше не нужны, немедленно удаляются. Это минимизирует пиковое использование дискового пространства.

## Тестовое покрытие

Юнит-тесты покрывают:
- Сортировку пустого файла, одного элемента, уже отсортированных и реверсированных данных
- Обработку дубликатов и отрицательных чисел
- Базовые операции FileTape (чтение, запись, перемещение)
- Корректность сортировки при различных лимитах памяти
